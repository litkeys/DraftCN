# Story 1.11: Canvas Zoom Control

## Status

Approved

## Story

**As a** user,
**I want** to adjust the zoom level of the canvas using a slider in the navigation bar,
**so that** I can zoom in for detailed work or zoom out to see the full design.

## Acceptance Criteria

1. Zoom control slider positioned in the center of the top navigation bar
2. Slider shows current zoom percentage (e.g., "100%") as a label next to the slider
3. Zoom range from 25% minimum to 200% maximum
4. Default zoom level is 100% (actually renders at 0.8 scale for better viewport fit)
5. Slider has discrete steps: 25%, 50%, 75%, 100%, 125%, 150%, 175%, 200%
   - 100% = 0.8 scale (default, fits more content)
   - 125% = 1.0 scale (true 1:1)
   - Scale calculation: actualScale = sliderValue \* 0.8
6. Canvas uses viewport/camera system: blocks scale proportionally by recalculating positions, dimensions, and font sizes based on zoom level (no CSS transforms)
7. Block previews during drag operations scale by applying zoom factor to preview dimensions and positioning
8. When zoomed in beyond container width, horizontal scrollbar appears at bottom of canvas container
9. Block drag and drop from library sidebar maintain accuracy at all zoom levels using camera-relative positioning
10. Block selection and movement on canvas maintain accuracy at all zoom levels using camera-relative positioning
11. All block properties (width, height, x, y, fontSize) dynamically calculated using: `scaledValue = originalValue * zoomFactor`
12. Text remains crisp at all zoom levels (no transform-induced blurriness)
13. No CSS transform stacking context issues with overlapping or wrongly positioned elements

### Implementation Approach

**Viewport/Camera System Implementation**:
The requirement specifies a viewport/camera system rather than CSS transforms. This means:

- All block positions, dimensions, and font sizes are recalculated based on zoom level
- No use of CSS `transform: scale()` which can cause blurriness and stacking context issues
- Direct calculation approach: `scaledValue = originalValue * actualScale`
- actualScale = zoomPercentage \* 0.8 (so 100% = 0.8 scale, 125% = 1.0 scale)

**State Management** [Source: architecture/frontend-architecture.md]:

- Use Zustand sliced architecture pattern
- Create new `ZoomSlice` at `store/slices/zoom.ts`
- Follow existing pattern from `store/slices/drag.ts` and `store/slices/blocks.ts`
- Export selectors separately for better organization

**Component Structure** [Source: architecture/components.md]:

- Canvas component at: `components/canvas/Canvas.tsx`
- Header component at: `components/layout/Header.tsx`
- Create new: `components/controls/ZoomControl.tsx`
- Grid system at: `lib/grid/calculator.ts`

**Existing Canvas Implementation**:

- Canvas uses direct block rendering (no CSS transforms currently)
- Blocks positioned with absolute positioning using inline styles
- Mouse event handlers already convert coordinates to canvas-relative
- Grid snapping function at `lib/grid/calculator.ts` needs zoom factor

**Coordinate Transformation Math**:

```typescript
// Convert mouse position to canvas coordinates accounting for zoom
const canvasX = (clientX - canvasRect.left) / actualScale
const canvasY = (clientY - canvasRect.top) / actualScale

// Scale block properties for rendering
const scaledBlock = {
  x: block.x * actualScale,
  y: block.y * actualScale,
  width: block.width * actualScale,
  height: block.height * actualScale,
  fontSize: (block.fontSize || 16) * actualScale,
}
```

### Important Implementation Notes

1. **Viewport System vs CSS Transform**: Must use viewport/camera calculations, NOT CSS transforms. This ensures:

   - Text remains crisp (no transform blur)
   - No stacking context issues
   - Accurate mouse event handling
   - Better performance for complex layouts

2. **Scale Factor**: The 0.8 factor means:

   - 100% zoom = 0.8 actual scale (fits more content)
   - 125% zoom = 1.0 actual scale (true 1:1)
   - Formula: `actualScale = zoomLevel * 0.8`

3. **Coordinate Systems**: Two coordinate systems in play:

   - **Canvas coordinates**: The logical positions (stored in blocks)
   - **Screen coordinates**: The rendered positions (after zoom)
   - Always store in canvas coordinates, render in screen coordinates

4. **Performance Considerations**:

   - Use React.memo on Canvas when zoom changes
   - Consider requestAnimationFrame for smooth zoom transitions
   - Batch block recalculations

5. **Drag Operation Accuracy**: Critical to maintain:

   - Library to canvas drag accuracy
   - Block repositioning accuracy
   - Selection rectangle accuracy
   - All must work correctly at any zoom level

6. **Grid System Integration** [FUTURE ENHANCEMENT]:
   - Grid system functionality is not yet implemented (Epic 2)
   - Future zoom-grid integration will require:
     - Grid background pattern scaling with zoom
     - SnapToGrid calculations accounting for zoom: `snapToGrid(x / actualScale, y / actualScale)`
     - Grid snapping accuracy at all zoom levels
     - Grid bypass (Alt key) compatibility with zoom
   - Defer grid integration until Epic 2 implementation is complete
