# Story 1.2: Block Data Model and Template System

## Status

Draft

## Story

**As a** developer,  
**I want** to implement the template-based block architecture,  
**so that** blocks have reusable templates with customizable instances.

## Acceptance Criteria

1. BlockTemplate interface includes (typeId, name, category, thumbnail, dependencies, defaultProps, component, defaultWidth/Height, minimumWidth/Height)
2. Block instance interface includes (id, typeId, props, x, y, width, height, z, selected)
3. Template registry system created to store and manage available templates
4. Manual template registration process defined with clear structure
5. At least 3 sample templates registered (hero1, navbar1, footer2) from Shadcnblocks References
6. Registry supports template registration, retrieval by typeId, and instance creation
7. All position values stored in pixels as specified

### Data Models
**[Source: architecture/data-models.md#Block]**
- Block interface requires: id (string), typeId (string), props (any), x/y/width/height (numbers in pixels), z (number for stacking), selected (boolean)
- Block instances reference templates via typeId
- Props must conform to template's specific interface

**[Source: architecture/data-models.md#BlockTemplate]**
- BlockTemplate requires: typeId, name, category, thumbnail, dependencies array, defaultProps, component (React.ComponentType), defaultWidth/Height, minimumWidth/Height
- Dependencies extracted from import statements
- Component is actual React component reference, not code string

**[Source: architecture/data-models.md#BlockRegistry]**
- Registry uses Map<string, BlockTemplate> for template storage
- Methods needed: registerTemplate(), getTemplate(), getTemplatesByCategory(), generateBlockInstance()
- Block instance generation creates unique IDs using timestamp + random string

### File Locations
**[Source: architecture/unified-project-structure.md]**
- Type definitions: `types/block.ts`, `types/template.ts`, `types/canvas.ts`
- Registry implementation: `lib/blocks/registry.ts`, `lib/blocks/types.ts`
- Template components: `templates/{name}.tsx`
- Thumbnails: `public/thumbnails/{typeId}.png`

### Component Specifications
**[Source: architecture/data-models.md#Methods for Block Template Development]**
- Templates are React functional components with typed props
- Each template has accompanying registration object with all metadata
- Manual registration process in central registry file
- Components reference global CSS classes from `app/globals.css`

**[Source: Shadcnblocks References]**
- Using existing shadcnblocks templates: hero1.tsx, navbar1.tsx, footer2.tsx
- These templates already have proper TypeScript interfaces and default props
- Templates use shadcn/ui components (Badge, Button, NavigationMenu, Sheet, etc.)
- Each template has comprehensive props for customization

**[Source: docs/adding-block-templates.md]**
- Components go in `templates/` directory as individual files
- Each template must be registered in `lib/blocks/registry.ts`
- Registration includes: typeId, name, category, thumbnail path, dependencies, defaultProps, component reference, dimensions
- TypeId naming convention: lowercase with hyphens (e.g., 'hero1', 'navbar1', 'footer2')
- Thumbnails go in `public/thumbnails/` with matching typeId names

### Technical Constraints
**[Source: architecture/tech-stack.md]**
- TypeScript 5.3+ with strict mode required
- React 19.0+ for component runtime
- Template management via JavaScript objects (manual registration)
- Global CSS for shared block styles
- No backend/database - all state in memory

### Testing Requirements

- Test files named with `.test.ts` suffix
- Use Vitest 1.0+ when configured (future story)
