# Story 1.3: Template Registration System

## Status

Skipped - Already Implemented

## Story

**As a** developer,  
**I want** to manually register block templates in the registry,  
**so that** templates are available for use in the builder.

## Acceptance Criteria

1. Clear template registration structure defined in registry file
2. Each template entry includes all required metadata
3. Component imports properly configured
4. Props interfaces clearly defined for each template
5. Default props specified for initial rendering
6. Templates organized by category for easy management
7. Registration process documented with examples

### Data Models

**[Source: architecture/data-models.md#BlockTemplate]**

BlockTemplate interface requirements:
```typescript
interface BlockTemplate {
  typeId: string;           // Unique identifier (e.g., 'hero-1')
  name: string;             // Display name (e.g., 'Hero Section')
  category: string;         // Category for grouping (e.g., 'Heroes')
  thumbnail: string;        // Path to preview image
  dependencies: string[];   // Required imports/packages
  defaultProps: any;        // Initial content for rendering
  component: React.ComponentType<any>;  // React component reference
  defaultWidth: number;     // Initial width in pixels
  defaultHeight: number;    // Initial height in pixels
  minimumWidth: number;     // Minimum allowed width
  minimumHeight: number;    // Minimum allowed height
}
```

**[Source: architecture/data-models.md#BlockRegistry]**

Registry interface already implemented with these methods:
- `registerTemplate(template: BlockTemplate): void` - Add template to registry
- `getTemplate(typeId: string): BlockTemplate | undefined` - Retrieve by typeId
- `getTemplatesByCategory(category: string): BlockTemplate[]` - Filter by category
- `generateBlockInstance(typeId: string, props?: any): Block` - Create instance

### File Locations

**[Source: architecture/unified-project-structure.md#templates]**

Correct structure for templates:
- `/templates/` - Root templates directory
- `/templates/{category}/` - Category subdirectories (hero/, navigation/, footer/)
- `/templates/{category}/{name}.tsx` - Template component files
- `/templates/{category}/index.ts` - Category exports
- `/templates/index.ts` - Central registration file
- `/public/thumbnails/{typeId}.svg` - Thumbnail images

Current structure needs reorganization to match architecture specification.

### Component Specifications

**[Source: architecture/data-models.md#Methods for Block Template Development]**

Manual registration pattern required:
```typescript
const heroTemplate: BlockTemplate = {
  typeId: 'hero-1',
  name: 'Hero Section',
  category: 'Heroes',
  thumbnail: '/thumbnails/hero-1.png',
  dependencies: ['@/components/ui/button'],
  defaultProps: { /* ... */ },
  component: HeroComponent,
  defaultWidth: 1200,
  defaultHeight: 400,
  minimumWidth: 600,
  minimumHeight: 200
};
registry.registerTemplate(heroTemplate);
```

**[Source: architecture/components.md#Block Registry]**

Registry serves as central repository - should be pure class without template imports.
Templates should be registered during initialization in separate file.

### Technical Constraints

**[Source: architecture/tech-stack.md#Template Management]**
- JavaScript Objects for template storage (no database)
- React.lazy for potential dynamic loading (React 19.0+)
- Manual registration process (no auto-discovery)
- Global CSS for shared styles

**[Source: architecture/coding-standards.md#Block Templates]**
- Templates must be manually registered with complete metadata
- TypeScript strict mode required
- Functional components only (no class components)
- Component naming: PascalCase
- File naming: kebab-case or PascalCase
- TypeId naming: lowercase with hyphens

### Testing Requirements

**[Source: architecture/testing-strategy.md#Frontend Tests]**

- Unit tests location: `/tests/unit/templates/registration.test.ts`
- Test framework: Vitest 1.0+ (when configured)
- Test file naming: `.test.ts` suffix
- Focus areas:
  - Template registration verification
  - Metadata completeness checks
  - Category organization validation
  - Error handling for duplicates

### Project Structure Notes

Current structure deviates from architecture specification:
- Templates currently flat in `/templates/` instead of organized by category
- Registration happening in `lib/blocks/registry.ts` instead of `templates/index.ts`
- No category-level index files for exports

This story will align the implementation with the documented architecture.

## Skipped Reason

**Decision Date:** 2025-09-09

**Reason:** All 7 acceptance criteria were already fully implemented in Story 1.2:

1. ✅ Clear template registration structure - Implemented in `lib/blocks/registry.ts` with detailed comments
2. ✅ Complete metadata for each template - All templates have typeId, name, category, thumbnail, dependencies, defaultProps, component, dimensions
3. ✅ Component imports properly configured - Hero1, Navbar1, Footer2 imported and working
4. ✅ Props interfaces clearly defined - Each template has TypeScript interfaces
5. ✅ Default props specified - Comprehensive realistic content provided
6. ✅ Templates organized by category - Categories defined and `getTemplatesByCategory()` method works
7. ✅ Registration process documented - Extensive inline documentation with examples

The proposed file reorganization (moving templates to category subdirectories and creating `templates/index.ts`) would:
- Break working production-ready code
- Add no functional value
- Create unnecessary refactoring work
- Potentially introduce bugs

**Current Implementation Decision:** Keep the existing manual registration in `lib/blocks/registry.ts` as it is clearer, working, and follows the manual registration pattern effectively.
